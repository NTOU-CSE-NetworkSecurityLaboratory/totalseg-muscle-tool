# 功能總覽

這份文件不是在講程式怎麼寫，而是在講這個工具目前到底有哪些功能。

目的只有兩個：

1. 先把產品功能講清楚
2. 再用這些功能去反推程式應該怎麼重構

這一版先不談 UI，也不談 webview，只看工具本身的能力。

## 1. 這個工具現在能做什麼

目前可以把整個工具理解成 4 個主要功能。

### 功能 1：對 CT 或 MRI 做肌肉分割

這是最核心的功能。

使用者提供一個 DICOM case，工具會跑 TotalSegmentator，產生對應 task 的 segmentation 結果。

這個功能的目的，是先回答一件事：

「這個 case 的每一張影像，哪些 pixel 屬於目標肌肉或目標結構？」

這個功能的輸入大致是：

- 一個 DICOM case
- modality，例如 CT 或 MRI
- task，例如 muscle 類或 spine 類
- 是否使用 fast 模式

這個功能的輸出大致是：

- segmentation 檔案
- statistics 資料
- 給後續 CSV 與 PNG 使用的基礎結果

### 功能 2：把分割結果整理成可看的數值

分割完之後，不只需要 mask，還需要能看懂的數值。

所以工具會把分割結果整理成 CSV，讓使用者可以看到：

- 每一張 slice 的面積
- 每一張 slice 的 HU 平均值
- 每一張 slice 的 HU 標準差
- 整體的 summary，例如 volume、pixel count、mean HU

這個功能的重點是：

「把 segmentation 轉成可以放進 Excel 或後續分析的表格結果」

這個功能最重要的正確性條件是：

- CSV 的 slice 順序必須跟原始 DICOM 順序一致
- CSV 的 slice 順序也必須跟後續 PNG 對得起來

### 功能 3：把分割結果畫回原始影像

除了數值，還需要肉眼確認分割到底畫在哪裡。

所以工具會把 segmentation 疊回 DICOM 影像，輸出成逐張 PNG。

這個功能的目的，是回答：

「這一張圖上面，模型到底把哪一塊當成肌肉？」

這個功能通常會產生：

- 一般 overlay PNG
- 視情況產生 erosion 後的 PNG

這個功能最重要的正確性條件是：

- 有 mask 的 slice，應該要有對應渲染結果
- 圖片順序要和 DICOM、CSV 一致

### 功能 4：把 AI 結果和人工標註做比較

這是另一條比較獨立的分析功能。

它會拿 AI 產生的 mask 和 manual mask 做比對，算出像是：

- Dice
- 面積差異

這個功能的目的，是回答：

「AI 的結果和人工標註差多少？」

這個功能和前面三個功能有關，但它比較像分析工具，不是主分割流程本身。

## 2. 現在的實際工作流程

如果只看核心功能，不看 UI，現在的流程大概是這樣：

1. 先讀取一個 DICOM case
2. 選擇 modality 與 task
3. 執行分割
4. 取得 segmentation 結果
5. 把結果整理成 CSV
6. 把結果畫成 PNG
7. 如果需要，再拿去和 manual mask 比較

所以本質上，這個工具不是很多散開的小工具，而是一條主流程加上一條比較流程：

- 主流程：DICOM -> segmentation -> CSV / PNG
- 比較流程：AI mask + manual mask -> comparison metrics

## 3. 目前應該怎麼理解每個功能的角色

如果從產品角度看，目前最核心的是前 3 個功能，而且它們是連在一起的。

### 分割

分割是源頭。

沒有分割結果，後面就不會有 CSV，也不會有 PNG。

### CSV

CSV 是把分割結果變成數值摘要。

它的價值在於：

- 可以做 Excel 檢查
- 可以做統計分析
- 可以快速看哪些 slice 有數值、哪些沒有

### PNG

PNG 是把分割結果變成視覺檢查材料。

它的價值在於：

- 可以用肉眼驗證 segmentation 是否合理
- 可以對照 CSV 看每一張 slice 的結果

### Compare

Compare 是驗證與研究用途。

它不是每次主流程都一定要跑，但它能幫助我們知道模型結果和人工標註之間的差距。

## 4. 目前產品層面的主要問題

如果不看程式碼，只看功能，現在最大的問題其實不是「功能太少」，而是「功能邊界還不夠清楚」。

主要有 4 類問題。

### 問題 1：分割、數值整理、圖片渲染的責任還有點混

現在你會直覺覺得 `seg` 和 `draw` 有點混在一起，這個感覺是對的。

原因不是它們不能分開，而是目前還沒有用產品語言明確規定：

- 分割功能做到哪裡算結束
- CSV 是不是分割功能的一部分
- PNG 是不是分割完自動附帶的功能

這些如果不先講清楚，程式就會越寫越纏在一起。

### 問題 2：輸出之間的對位非常重要，但目前還不夠明確

這個工具有三種核心輸出：

- segmentation
- CSV
- PNG

它們其實都在描述同一批 slice。

所以產品上必須保證：

- 第 N 張 DICOM
- CSV 的第 N 筆 slice
- PNG 的第 N 張結果

是在說同一件事。

之前你遇到「Excel 看起來只有最後一張有數值」這種問題，本質上就是這個對位不夠穩。

### 問題 3：批次處理是功能，但現在還沒有被當成獨立功能定義清楚

工具其實不只是在跑單一 case，也常常會跑多個 case。

所以「批次處理」本身應該被當成功能看待，而不是附帶細節。

它至少要講清楚：

- 怎麼掃描病例
- 哪些資料夾算 case
- 哪些資料夾要略過
- 單一 case 失敗時要不要繼續

### 問題 4：錯誤與診斷能力還不夠產品化

現在有些錯誤能看懂，有些還不夠清楚。

但從產品角度來看，工具除了要會跑，還要能回答這些問題：

- 是哪一個 case 失敗
- 是 segmentation 沒出來，還是 CSV 沒寫出來
- 是真的只有最後一張有 mask，還是順序對錯了

這些都應該算功能的一部分，不只是工程內部細節。

## 5. 我們理想上想把產品整理成什麼樣子

如果先不談 UI，只談功能，我會建議把產品最後整理成下面這種理解方式。

### 核心功能 A：分割流程

負責把 DICOM 變成 segmentation 結果。

### 核心功能 B：數值輸出流程

負責把 segmentation 變成 CSV 與 summary。

### 核心功能 C：影像輸出流程

負責把 segmentation 疊回 DICOM，輸出 PNG。

### 核心功能 D：比較流程

負責把 AI mask 與 manual mask 做比較。

### 核心功能 E：批次執行流程

負責把上面幾個流程套用到多個 case。

### 核心功能 F：診斷與錯誤回報

負責明確告訴使用者每一步有沒有成功、失敗在哪裡、輸出是否對位正常。

這樣整理之後，產品功能就會比較清楚，不會一直卡在「某個檔案到底該不該做這件事」。

## 6. 之後程式應該怎麼跟著這些功能重構

這裡才開始談重構，而且只談方向。

重構順序應該是：

1. 先把功能定義講清楚
2. 再讓 CLI 成為這些功能的唯一正式入口
3. 再把 GUI 改成只負責操作這些功能
4. 最後再把 webview 改成同樣的薄殼

也就是說，之後的程式不是先想：

- `seg.py` 該不該呼叫 `draw.py`

而是先想：

- 分割功能的正式輸出是什麼
- 數值輸出功能是不是獨立功能
- 圖片輸出功能是不是獨立功能
- 批次執行是不是另一層功能

先把這些定義清楚，程式自然比較容易拆。

## 7. 接下來最應該先做什麼

現在最重要的不是立刻大改程式，而是先把這份文件逐條對準真實功能。

建議下一步就是逐條確認這幾件事：

1. 目前真正的主功能是不是這 4 類
2. CSV 應不應該被視為分割後必定產生的正式輸出
3. PNG 應不應該被視為分割後可選的影像輸出
4. Compare 是否維持獨立功能
5. 批次處理是否要升級成正式功能
6. 診斷與錯誤訊息是否要被當成正式功能需求

如果這些先對齊，後面的 CLI 重構、GUI 薄化、webview 薄化才會做得穩。
